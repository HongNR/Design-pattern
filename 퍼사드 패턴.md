# 퍼사드 패턴 (Facade Pattern)

## 퍼사드 패턴이란?

> **복잡한 서브시스템에 대해 단순화된 통합 인터페이스를 제공하는 구조 패턴**

'Facade'는 건물의 정면(외벽)을 의미합니다. 내부의 복잡한 구조는 숨기고, 밖에서 보기에는 깔끔하고 사용하기 쉬운 **하나의 창구**를 만들어주는 역할을 합니다.

이 패턴은 시스템이 복잡해질수록 진가를 발휘하며, 클라이언트가 수많은 클래스와 상호작용하는 대신 단 하나의 객체(Facade)와만 소통하게 하여 시스템 간의 의존성을 획기적으로 낮춰줍니다.

## 구조

```
┌─────────────┐
│  Client     │
└──────┬──────┘
       │
       ▼
┌─────────────────────────────────┐
│           Facade                │
│ (단순화된 인터페이스 제공)          │
└──────┬──────────┬──────────┬────┘
       │          │          │
       ▼          ▼          ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│ Subsystem│ │ Subsystem│ │ Subsystem│
│    A     │ │    B     │ │    C     │
└──────────┘ └──────────┘ └──────────┘
```

### 주요 구성 요소

- **Facade**: 클라이언트의 요청을 적절한 서브시스템 객체로 전달하는 단순화된 인터페이스.
- **Subsystem Classes**: 복잡한 기능을 수행하는 여러 클래스들. Facade의 존재를 모르며 직접적으로 상호작용함.
- **Client**: 서브시스템을 직접 호출하는 대신 Facade를 통해 상호작용함.

## 코드 예시 1 (홈 시어터 시스템)

영화 한 편을 보려면 TV를 켜고, 사운드바를 조절하고, 플레이어를 실행하고, 조명을 조절해야 합니다. 이 복잡한 과정을 퍼사드 패턴으로 단순화해 보겠습니다.

### 1. 복잡한 서브시스템들

```java
class TV {
    void on() { System.out.println("TV를 켭니다."); }
    void off() { System.out.println("TV를 끕니다."); }
}

class SoundBar {
    void on() { System.out.println("사운드바를 켭니다."); }
    void setVolume(int level) { System.out.println("사운드바 볼륨을 " + level + "로 설정합니다."); }
    void off() { System.out.println("사운드바를 끕니다."); }
}

class Lights {
    void dim() { System.out.println("조명을 어둡게 조절합니다."); }
    void on() { System.out.println("조명을 밝게 켭니다."); }
}

class MoviePlayer {
    void play(String movie) { System.out.println(movie + " 영화를 재생합니다."); }
    void stop() { System.out.println("영화를 중지합니다."); }
}
```

### 2. 퍼사드 (HomeTheaterFacade)

```java
public class HomeTheaterFacade {
    private TV tv;
    private SoundBar soundBar;
    private Lights lights;
    private MoviePlayer player;

    public HomeTheaterFacade(TV tv, SoundBar soundBar, Lights lights, MoviePlayer player) {
        this.tv = tv;
        this.soundBar = soundBar;
        this.lights = lights;
        this.player = player;
    }

    // 복잡한 과정을 하나의 메서드로 단순화
    public void watchMovie(String movie) {
        System.out.println("--- 영화 관람 준비 중 ---");
        lights.dim();
        tv.on();
        soundBar.on();
        soundBar.setVolume(10);
        player.play(movie);
        System.out.println("--- 관람 시작 ---");
    }

    public void endMovie() {
        System.out.println("--- 영화 관람 종료 중 ---");
        player.stop();
        soundBar.off();
        tv.off();
        lights.on();
        System.out.println("--- 종료 완료 ---");
    }
}
```

### 3. 패턴 적용 전 vs 후 비교

#### [Before] 퍼사드 패턴 미적용 (클라이언트가 모든 기기를 직접 제어)
클라이언트가 TV, 사운드바, 조명 등의 클래스를 모두 알고 있어야 하며, 실행 순서도 직접 관리해야 합니다.

```java
public class NonFacadeClient {
    public static void main(String[] args) {
        TV tv = new TV();
        SoundBar sb = new SoundBar();
        Lights lights = new Lights();
        MoviePlayer player = new MoviePlayer();

        // 영화 한 편 보려는데 코드가 이렇게나 많이 필요합니다...
        System.out.println("--- 영화 관람 준비 중 ---");
        lights.dim();
        tv.on();
        sb.on();
        sb.setVolume(10);
        player.play("인셉션");
        
        // 종료할 때도 똑같은 고생을 반복해야 합니다.
    }
}
```

#### [After] 퍼사드 패턴 적용 (클라이언트는 퍼사드만 호출)
클라이언트는 복잡한 기기들을 알 필요 없이, 퍼사드가 제공하는 '간편한 버튼' 하나만 누르면 됩니다.

```java
public class FacadeClient {
    public static void main(String[] args) {
        // 퍼사드 객체 하나만 준비합니다.
        HomeTheaterFacade homeTheater = new HomeTheaterFacade(new TV(), new SoundBar(), new Lights(), new MoviePlayer());

        // 딱 한 줄로 모든 준비가 끝납니다!
        homeTheater.watchMovie("인셉션");
        
        System.out.println("\n... 영화 관람 중 ...\n");
        
        // 끌 때도 한 줄이면 충분합니다.
        homeTheater.endMovie();
    }
}
```

## 코드 예시 2 (실제 프로젝트: TastUtopia)

**TastUtopia**에서도 퍼사드 패턴의 원리가 서비스 계층에서 활발히 사용되고 있습니다. 특히 `ReviewService`는 여러 도메인(사용자, 가게, 리뷰)을 하나로 묶어 처리하는 퍼사드 역할을 수행합니다.

### ReviewService (Facade 역할)

```java
@Service
@RequiredArgsConstructor
public class ReviewService {

    private final ReviewSpringJpaRepository reviewRepository;
    private final UserService userService;   // 서브시스템 1
    private final StoreService storeService; // 서브시스템 2

    @Transactional
    public Long saveReview(ReviewRequestDto request) throws Exception {
        Store store = storeService.findByStoreSeq(request.getStore());
        
        User user = userService.findUserBySeq(request.getUser());
        
        Review review = new Review(store, user, request.getComment(), ...);
        reviewRepository.save(review);

        return review.getSeq();
    }
}
```

**효과**: `ReviewController`는 리뷰 하나를 저장하기 위해 `UserService`나 `StoreService`를 직접 알 필요가 없습니다. 오직 `ReviewService`의 `saveReview`라는 단순화된 창구만 이용하면 됩니다.

## 장단점

### 장점

1. **시스템의 복잡성 숨김**
   - 퍼사드 패턴을 사용하면, 개발자는 복잡한 서브시스템의 내부 로직을 몰라도 되고, 단순화된 인터페이스를 통해서만 시스템과 상호작용할 수 있다. 이렇게 되면 개발 과정이 단순해지고, 오류 가능성도 줄어든다.

2. **의존성 감소**
   - 클라이언트 코드는 서브시스템의 구체적인 구현에 의존하지 않고, 단지 퍼사드 인터페이스에만 의존하게 된다. 이런 방식은 시스템의 결합도를 낮추어 유지보수와 확장성을 높이는 데 도움을 준다.

3. **유지보수 및 확장 용이**
   - 서브시스템의 변경이 있어도 퍼사드 인터페이스는 그대로 유지될 수 있다. 이것은 시스템을 업그레이드하거나 수정할 때, 클라이언트 코드에 미치는 영향을 최소화해주는 장점이 있다.

### 단점

1. **퍼사드의 과도한 책임**
   - 퍼사드가 서브시스템의 모든 요소를 추상화하려고 하면, 너무 많은 책임과 복잡성을 가지게 될 수 있다. 이는 퍼사드 자체가 복잡해지고 관리하기 어려워지는 원인이 될 수 있다.

2. **서브시스템에 대한 의존성**
   - 퍼사드는 서브시스템에 대해 어느 정도 의존성을 가질 수밖에 없다. 만약 서브시스템이 크게 변경된다면, 퍼사드 역시 이에 맞춰 수정되어야 하며, 이는 유지보수의 복잡성을 증가시킬 수 있다.

3. **유지보수의 복잡성**
   - 퍼사드가 제공하는 추가적인 추상화 레이어는 코드 베이스에 추가적인 계층을 만든다. 이렇게 되면 간단한 작업조차도 퍼사드를 통해 이루어져야 하고, 이는 유지보수를 더 복잡하게 만들 수 있다. 특히, 시스템이 커지고 복잡해질수록 이러한 문제는 더 심각해질 수 있다.

## 추가로 알면 좋은 내용

### 1. 최소 지식 원칙 (Principle of Least Knowledge)
- 퍼사드 패턴은 '데메테르의 법칙(Law of Demeter)'을 실현하는 대표적인 방법입니다.
- "낯선 이와 대화하지 말고 가까운 친구(Facade)하고만 대화하라"는 원칙을 통해 클라이언트가 서브시스템의 깊숙한 내부까지 알지 못하게 차단합니다. 이는 코드의 가독성과 유지보수성을 극대화합니다.

### 2. 관련 패턴 비교
- **Adapter**: 인터페이스를 **변환**하여 호환되지 않는 두 객체를 연결.
- **Facade**: 인터페이스를 **단순화**하여 사용하기 편하게 만듦.

### 실제 사용 사례
- **SLF4J**: 다양한 로깅 라이브러리를 하나의 인터페이스로 통합.
- **Spring Templates**: `RestTemplate`, `JdbcTemplate` 등 복잡한 연결/해제 과정을 단순화.

## 핵심 정리

```
퍼사드 패턴 = 복잡한 시스템의 '통합 안내 데스크'

[클라이언트] -> [안내 데스크(Facade)] -> [복잡한 부서들(Subsystems)]

- 목적: 인터페이스 단순화, 결합도 완화
- 핵심: 내부의 복잡성은 숨기고, 외부에는 편의성만 제공한다!
```
